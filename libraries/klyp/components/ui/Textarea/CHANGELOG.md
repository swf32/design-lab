# Textarea — changelog

## 2026-06-30 04:48 — merge-autotextarea-autogrow-grip-free-default

- What: Merged the standalone `AutoTextarea` primitive INTO Textarea behind a new `autoGrow` prop (+ `minRows`/`maxRows`/`onValueChange`/`textareaRef`): with `autoGrow`, JS measures `scrollHeight` and sets an explicit height clamped to `minRows…maxRows` (cross-browser grow + a hard max-rows cap), and `[data-autogrow]` neutralises the per-size min-height floor + CSS `field-sizing`. Default `resize` flipped `vertical` → `none` so the native `::-webkit-resizer` grip no longer clips the rounded `--r-chip` corner (the field auto-grows via `field-sizing` / `autoGrow` instead); the `resize` prop stays for opt-in. Deleted `packages/ui/src/AutoTextarea` (folder + barrel export + `auto-textarea` registry entry). Consumers migrated to `<Textarea autoGrow>`: composer-multishot, series episode (synopsis / scenario / scene-description), asset-compose, asset-metadata-editor, cast-screen — their bespoke field SCSS (border / radius / bg / focus / resize) stripped, now owned by the DS base.
- Why: AutoTextarea (JS scrollHeight grow) and Textarea (CSS `field-sizing`) were duplicate auto-grow engines — unify on one primitive. JS `autoGrow` keeps cross-browser growth + the max-rows ceiling `field-sizing` lacks (no Firefox / old-Safari support). Grip-free default fixes the ugly native resizer overflowing the rounded corner.

## 2026-06-25 14:13 — size-resize-props-and-invalid-cleanup

- What: Added `size` (sm/md/lg → `[data-size]`) and `resize` (none/vertical/horizontal/both → `[data-resize]`, default `vertical`) props, mirroring Input. Size ramp follows Input EXACTLY: only `sm` shrinks the font (`--type-body-sm`); `md` and `lg` keep base `--type-body` — size changes the box (min-height floor 48/64/80 + padding), not the kerning. Padding rebalanced so the text isn't in a wide letterbox: horizontal mirrors Input per size (8/12/12), vertical kept close (6/10/12) instead of the old 6/12. Radius now uses semantic `--r-chip` (was raw `--radius-md`); hardcoded `4rem` min-height tokenised to `--space-64`. Fixed the "two focus borders" bug: a bare native `<textarea>` has no `data-rac`, so the global `:focus-visible:where(:not([data-rac]))` ring in globals.scss painted a second offset outline over our box-shadow ring — added `outline: none` inside `.klyp-Textarea:focus-visible` ((0,2,0) beats the global (0,1,0)), leaving the single Input/SearchField-style ring. Removed the dead `[data-invalid]` selector — invalid via hand-set `aria-invalid` only (same as Input). Stories reworked to mirror Input: `Sizes`, `Resize`, `Adaptive`, and two composition stories (`label + helper`, `char counter`). Consumers `create-episode-modal`, `promote-to-asset-modal`, `scene-create-form` migrated off `&__textarea` SCSS overrides onto the `resize` prop / defaults. State parity with Input/SearchField: added the missing `:hover` border (`--color-border-strong`), tokenised disabled `opacity: 0.5` → `var(--opacity-disabled)`, and flipped readonly from `--color-fg-muted` to `--color-fg-primary` (readonly value is content to read/copy — full contrast, matching Input).
- Why: cross-DS audit (vs Radix/HeroUI/Geist/React Aria + sibling Input/SearchField) — 2/3 live consumers fought the hardcoded `resize`, no size axis the others ship, radius drifted from Input, `[data-invalid]` shipped dead. First pass got three details wrong (invented a bigger `lg` font, 6/12 letterbox padding, and missed that a non-RAC textarea re-triggers the global focus ring) — corrected here against the Input/SearchField эталон. Textarea stays a primitive — no label/error props (lead's call): anatomy is composition, shown in stories like Input.
