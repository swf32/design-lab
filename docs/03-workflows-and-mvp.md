# Онбординг

Начало работы должно зависеть от ситуации пользователя.

Например:

- уже есть дизайн-система;

- есть только Figma;

- есть React-проект;

- проект создается с нуля.

Каждый сценарий приводит пользователя по собственному пути.

## Работа с AI

Практически в каждом разделе должны существовать:

- инструкция человеку;

- готовый промпт;

- возможность быстро скопировать этот промпт.

## Проверка полноты

После создания сущности Design Lab может автоматически предложить:

- создать Preview;

- добавить Stories;

- добавить документацию;

- создать примеры использования.

Но пользователь никогда не обязан это делать.

## История изменений

Интересна идея хранить историю продуктовых решений, а не только Git-историю.

Однако это считается сложной архитектурной задачей и пока не входит в ближайший MVP.

## Figma

На первом этапе Figma рассматривается только как источник контекста.

Импорт компонентов через плагины не является целью MVP.

Предполагается использование нейросетей для переноса между Figma и Design Lab.

В будущем Design Lab должен стать основным источником истины, а Figma — вспомогательным инструментом.

## Нативные платформы

Проект не должен ограничиваться Web.

В перспективе желательно поддерживать:

- SwiftUI;

- Android Compose;

- macOS;

- Windows.

Даже если визуализация будет приблизительной, Design Lab должен уметь работать с такими компонентами как с сущностями дизайн-системы.

## Аналитика

Wireframes и Pages должны быть удобны для передачи аналитикам.

Идеальный сценарий:

проект разворачивается на сервере, аналитики открывают нужный маршрут через браузер и проводят UX-исследования без необходимости запускать локальный репозиторий.

## Командная работа (будущее)

В перспективе рассматриваются:

- комментарии;

- заметки;

- совместная работа;

- синхронизация локальной версии с опубликованной;

- уведомления об изменениях;

- взаимодействие дизайнеров, аналитиков и разработчиков.

## MVP

На текущий момент в MVP входят:

- локальное веб-приложение;

- локальный Node.js runtime: dev server / API для файловой системы, watcher и индексатор;

- файловая система как источник истины;

- автоматическое обнаружение сущностей;

- Components;

- Pages;

- Wireframes;

- Assets;

- Palette;

- Tokens;

- Fonts;

- Rules;

- Storybook-подобный Playground;

- AI-промпты;

- автоматический индекс;

- семантический поиск;

- Embeddings;

- базовый MCP.

Не входят в MVP:

- глубокая интеграция с Figma;

- полноценная история продуктовых решений;

- сложная командная работа;

- роли пользователей;

- сложная серверная инфраструктура.

## Главная мысль проекта

**Design Lab — это не еще один инструмент для генерации интерфейсов. Это рабочая лаборатория, в которой цифровой продукт развивается как управляемая система. Код, компоненты, дизайн, документация, аналитика и нейросети работают вокруг одного источника истины — репозитория проекта.** 

По обзору рынка у Design Lab нет прямого аналога “один-в-один”. Ближайшие инструменты закрывают лишь части пространства: Storybook силён в изолированной разработке и документации компонентов; Bit — в компонентных workspace, composability и AI-native DX; Backstage — в каталоге сущностей и plugin platform; Builder.io, UXPin Merge, Plasmic и Anima — в визуальной сборке и design-to-code; Chromatic — в visual testing и review; zeroheight — в design-system documentation, measurement и MCP-контексте для AI. Из этого следует удачная ниша Design Lab: **не просто документация, не просто canvas и не просто design-to-code, а локальная продуктовая лаборатория для дизайнерской системы как набора файлов и сущностей**. [\[1\]](https://storybook.js.org/?utm_source=chatgpt.com)

Рекомендуемая продуктовая рамка для **MVP** такая. Базовый режим — **designer-first canonical workspace**: папка приложения `design-lab/` лежит рядом с `projects/` и `libraries/`. Внутри Project/Library хранятся артефакты по обязательному контракту Design Lab, а production-код туда не смешивается. Готовая дизайн-система подключается через миграцию в этот контракт; режим настройки произвольных директорий существующего репозитория не поддерживается.

Главные решения для MVP:

- **Rules включить в MVP**, в Markdown-first формате, потому что без них AI и проверки теряют опору.
- **Embeddings и поиск включить в MVP**, иначе AI будет слепым и не сможет собирать корректный контекст.
- **Локальный read-only MCP включён ранним adapter-слоем** поверх внутреннего context gateway. Он не содержит собственной бизнес-логики и имеет CLI fallback над тем же ядром. Remote transport, auth и write tools остаются за пределами MVP-среза. Это сохраняет внутреннюю согласованность файлового слоя и уже даёт coding agents проверяемый доступ к контексту. [\[2\]](https://zeroheight.com/mcp/?utm_source=chatgpt.com)
- **Figma в MVP — как context, а не как официальный importer/plugin**. Builder, Anima и похожие продукты активно строят Figma→code pipelines, но ваш продукт не должен повторять их тяжелую синхронизацию в первой фазе; разумнее начать с AI-driven ingest через ссылку, изображения и описание. [\[3\]](https://www.builder.io/c/docs/builder-figma-plugin?utm_source=chatgpt.com)
- **Storybook import policy — read-only / opportunistic**: если можно недорого импортировать stories и docs как контекст либо как стартовую документацию компонента — хорошо; если это начинает диктовать архитектуру продукта, импорт надо ограничить. Storybook и Chromatic стоит рассматривать как соседние системы, а не как базовый формат Design Lab. [\[4\]](https://storybook.js.org/docs/get-started/whats-a-story?utm_source=chatgpt.com)
- **Роли и ACL в локальном продукте не нужны**: локальная файловая система не делает такую модель надёжной. Ролевую модель имеет смысл обсуждать только для будущего hosted-слоя.

Итоговая рекомендация: **MVP должен доказать один главный цикл** — дизайнер создаёт или открывает проект, подключает библиотеку, использует токены/палитру/ассеты/шрифты, собирает компоненты, делает несколько wireframe-вариантов, выбирает page, проверяет правила, использует AI для ускорения, видит инспектор/пипетку, находит всё через embeddings-поиск, фиксирует changelog/decisions и экспортирует результат. Всё остальное — hosted-collab, глубокая аналитика, remote/write MCP, сложные импортеры и impact analysis — лучше оставить на следующие фазы.
